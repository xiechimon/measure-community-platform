# 一期后端全量上线与 TDD 交付设计

> 日期：2026-07-18
> 状态：已确认
> 适用范围：`docs/requirements/backend-requirements.md` 定义的一期 Java 后端全部 42 项能力

## 1. 背景

当前仓库已经完成 P0、P1 和 P2a：微服务骨架、统一响应与异常、OpenAPI 契约、人口黄金样板、RBAC 基础及功能级鉴权已经存在。正式需求基线中仍有 39 项“未开始”和 3 项“部分完成”，没有任何一项达到完整验收；`community-portal`、`community-service`、`community-welfare`、`community-affairs` 仍是 ping 空壳。

现有 45 个单元测试能够离线通过，但测试集中在 `community-common`、gateway、auth 和人口样板。MySQL、Redis、Nacos、gateway 到业务服务的真实链路尚未形成自动端到端验收，数据库迁移、敏感字段密文往返、RBAC 登录、AOP 权限和故障恢复仍存在上线风险。

首个上线版本不得裁剪正式需求范围，也没有固定截止日期。交付优化目标是：在严格执行 TDD 的前提下，尽快使全部 42 项能力达到可上线状态。

## 2. 目标与非目标

### 2.1 目标

- 正式需求基线中的 42 项能力全部达到“已完成”，并具备可重复的验收证据。
- 所有新增功能和缺陷修复执行可审计的 Red-Green-Refactor。
- 建立单元、集成、契约、端到端、安全、性能和恢复测试组成的发布门禁。
- 在真实 MySQL、Redis、Nacos 和 gateway 链路下验证数据正确性、权限、安全及跨服务可靠性。
- 形成可部署、可观测、可回滚、可备份恢复的后端发布物和运行手册。

### 2.2 非目标

- 不纳入 AI 侧和三级驾驶舱。
- 不新建与正式需求基线重复的范围台账。
- 不为追求覆盖率数字测试空壳、生成代码或无业务价值的访问器。
- 不在业务实现前建设超出当前规模需要的复杂测试平台或基础设施平台。
- 不以一次性“大爆炸”提交完成模块，也不以单独建表、空接口或仅有 Mock 测试证明需求完成。

## 3. 核心决策

采用“生产门禁先行、按依赖波次交付、全程严格 TDD”的推进方式。

TDD 是每个需求切片的开发门禁，系统级验收是上线门禁。二者互不替代：测试先行保证行为设计和回归安全，真实环境、安全、性能、可观测性及恢复演练保证可上线性。

不单独进行全仓库测试补写运动。既有代码只在即将修改或被新流程依赖时补特征测试；新增行为从第一行实现开始执行 TDD。这样可以把测试投入直接转化为需求完成度，并减少对临时代码的维护。

## 4. 需求与交付单元

### 4.1 唯一范围来源

`docs/requirements/backend-requirements.md` 中的稳定需求编号是唯一范围和完成度来源。路线图用于表达阶段依赖，不作为业务验收替代品。

每个需求编号拆为若干可独立验收的垂直切片。一个切片必须同时包含：

- OpenAPI 契约、请求/响应模型及示例；
- 数据库迁移、兼容策略和必要的回滚说明；
- 领域状态机、权限、数据范围、字段保护及审计；
- 领域单元测试和所需的基础设施集成测试；
- 至少一个从 API 边界验证验收条件的自动测试；
- 配置、可观测性和故障处理；
- 写回正式需求矩阵的源码、测试和验收命令证据。

### 4.2 完成定义

需求编号只有满足以下全部条件才可标记为“已完成”：

1. 该编号覆盖的全部验收切片均完成，不存在用后续任务兜底的核心路径。
2. 相关单元、集成、契约和 API 验收测试通过。
3. 真实依赖环境下的主路径、权限拒绝和关键失败路径通过。
4. 数据迁移、配置、日志、指标、告警和运行说明齐全。
5. 需求矩阵已记录可定位的实现与验收证据。
6. 不存在影响该需求上线的高风险已知缺陷。

## 5. 交付波次

### 5.1 Wave 0：上线验证基座

Wave 0 在继续 P2b 和业务铺开前完成，消除当前无法证明真实可运行的风险：

- 使用 Docker Compose 可重复启动 MySQL、Redis、Nacos 及最小应用集合。
- 从空库执行全部迁移和种子数据，修复 admin 占位 BCrypt，验证真实登录。
- 建立 gateway → auth/info → MySQL/Redis 的冒烟链路。
- 验证 Token、内部请求头、权限拒绝、人口数据密文落库和解密读取。
- 将 JWT、内部通信、AES、HMAC 等密钥外置；生产配置缺失时启动失败，不回退开发默认值。
- 建立存活、就绪和依赖健康检查，以及包含 traceId、用户、服务和错误码的日志规范。
- 建立 Maven 分层测试命令和 CI 门禁，确保本地与 CI 使用同一验收入口。
- 记录首发运行环境和代表性工作负载，冻结可重复的容量测试场景及发布阈值。

Wave 0 完成后，任何新增需求都必须通过相应自动门禁，不再只依靠离线 Mock 测试。

### 5.2 Wave 1：横切安全和公共能力

先完成所有业务域都会依赖的能力，避免后续逐模块回填：

- P2b：组织/网格行级数据范围、字段级脱敏、Refresh Token、角色管理和网关接口权限。
- X-03/X-04：隐私、敏感导出和统一审计。
- X-05：站内消息、通知偏好和可靠投递。
- X-06：附件授权、短期签名、下载审计和导出策略。
- X-07：外部凭证、签名验签、幂等、防重和告警公共约束。

### 5.3 Wave 2：信息与居务源服务

- 完成 `community-info` 的 I-01 至 I-07，包括人口完整字段、特殊人群、网格空间、车辆、资源、生态链和综合档案。
- 完成 `community-affairs` 的 A-01 至 A-09，包括活动、工单、诉求、场地、工具、公告、积分、设备和就业。

这两个服务先行，因为后续门户、居民端、物业端和服务订单都依赖其档案、工单、积分及告警数据。

### 5.4 Wave 3：服务交易与公益

- 完成 `community-service` 的 S-01 至 S-04，重点验证订单、支付、退款、履约、库存、核销及积分联动。
- 完成 `community-welfare` 的 W-01 至 W-03，包括资助审批、助农报表和法律援助外链台账。

涉及支付、积分、库存和审批的状态机必须先覆盖重复请求、回调乱序、超时、补偿及人工处理路径。

### 5.5 Wave 4：聚合和客户端 API

- 完成 `community-portal` 的 P-01 至 P-03，只读聚合已稳定的源业务服务。
- 完成 C-01 至 C-09 的居民端和物业端 API，复用源服务状态机，不复制业务规则或源数据。

### 5.6 Wave 5：全量发布验收

- 逐项关闭 42 个正式需求编号并审查证据。
- 执行全链路回归、安全测试、容量测试和故障注入。
- 演练数据库备份恢复、迁移、应用回滚和依赖故障降级。
- 审查生产配置、密钥、账号、网络暴露、日志脱敏、告警和运行手册。

## 6. TDD 执行协议

### 6.1 Red-Green-Refactor

每个可观察行为固定执行：

1. 从验收条件选择一个最小行为。
2. 写测试并运行，确认它因目标行为尚未实现而失败，而非环境或测试自身错误。
3. 写使该测试通过的最小生产代码。
4. 运行相关测试并确认转绿。
5. 在绿灯状态下重构，再次运行相关测试。
6. 完成该切片需要的集成或端到端测试。
7. 以包含需求编号的小粒度 Conventional Commit 提交。

失败测试阶段通常不提交。任务验收记录保存测试命令、预期失败原因和最终转绿结果，使 TDD 过程可审计。每个提交必须保持其作用域内测试通过。

### 6.2 强制规则

- 没有先失败的行为测试，不写对应业务实现。
- Bug 必须先用失败测试稳定复现。
- 修改未覆盖的既有代码前，先写特征测试锁定仍需保留的行为。
- Mock 只隔离外部边界，不得 Mock 掉待验证的核心领域逻辑、数据权限或事务行为。
- 覆盖率用于发现遗漏和防止回退，不作为 TDD 成功的单一证明。
- 生成 DTO、简单配置映射和无逻辑访问器不强求逐行单测，但其契约及运行效果必须由上层测试覆盖。
- SQL、配置、序列化、AOP、过滤器和消息行为使用集成测试证明，不以“难以单测”为由跳过自动验证。

### 6.3 测试分层

| 层级 | 主要范围 | 技术与约束 |
| --- | --- | --- |
| 领域单元测试 | 状态转换、校验、计算、权限判断、脱敏规则 | JUnit/Mockito；快速、无网络和数据库 |
| Web 切片测试 | 请求校验、响应、错误码、Controller 边界 | MockMvc/WebTestClient；不加载无关上下文 |
| 基础设施集成测试 | Mapper、迁移、事务、数据权限、Redis、幂等 | 真实 MySQL/Redis 容器；禁止用 H2 替代 MySQL 语义 |
| 契约测试 | OpenAPI 与实现、服务间事件/调用模型 | 生成校验加消费者/提供者契约 |
| 服务集成测试 | 服务内完整状态机和外部适配边界 | 启动单服务及必要真实依赖，外部第三方使用可控 stub |
| 端到端测试 | 登录、gateway、跨服务核心用户闭环 | Docker Compose 环境；经过真实路由和鉴权 |
| 非功能测试 | 安全、容量、故障、恢复 | 固定场景、数据集和环境说明，结果可重复比较 |

## 7. 服务边界与数据流

业务数据保持单一写入归属：

- `community-auth`：身份、权限、组织范围、隐私、审计、消息和安全策略。
- `community-info`：人口、房屋、车辆、资源及关联档案。
- `community-affairs`：活动、工单、诉求、场地、工具、公告、积分、设备和就业。
- `community-service`：服务项目、订单、支付、履约、库存和核销。
- `community-welfare`：资助、助农和法律援助。
- `community-portal`：只读聚合，不拥有或修改源业务状态。

跨服务写操作禁止直接共享表或用两个本地数据库事务模拟原子双写。订单—积分、业务事件—消息等流程使用本地事务加 outbox：

1. 源服务在一个本地事务内更新业务状态并追加 outbox 事件。
2. 投递器发布事件并记录投递结果。
3. 消费者以全局业务号和事件 ID 幂等处理。
4. 可恢复失败有限重试，超过阈值进入可查询、可重放的人工处理状态。
5. 补偿命令可重复执行并保留完整审计。

门户聚合读取失败时返回明确的部分失败或降级信息，不使用零值或缓存旧值伪装成实时完整数据。

## 8. 错误、安全与发布兼容

- 参数错误、业务冲突、认证失败、权限不足和系统错误使用稳定分段错误码；HTTP status 与响应语义一致。
- 未知异常只记录服务端诊断信息，对外不泄漏 SQL、堆栈、密钥或个人信息。
- 支付、对象存储、设备和其他外部调用设置明确超时、有限重试和熔断；非幂等调用禁止盲目重试。
- 每个状态机定义允许转换、重复请求结果、乱序处理和异常恢复路径。
- 数据库变更采用“先扩展、再切换、后清理”，在所有实例完成切换前不执行破坏性迁移。
- 发布包包含应用回滚步骤；涉及不可逆数据变更时必须先完成备份恢复演练。
- 生产环境缺少必需密钥、数据库连接或安全配置时快速失败，不使用开发默认值静默启动。
- 日志和指标携带 traceId、用户及资源上下文，但敏感字段只记录脱敏值或不可逆标识。

## 9. CI 与发布门禁

CI 使用递进门禁，失败立即停止后续阶段：

1. **快速门禁**：编译、静态检查、单元测试、OpenAPI 校验和数据库迁移静态审查。
2. **集成门禁**：真实 MySQL/Redis 下的迁移、数据访问、事务、权限、幂等及加密测试。
3. **系统门禁**：启动核心服务，执行经过 gateway 的身份、权限和核心业务端到端测试。
4. **发布门禁**：依赖与安全扫描、固定场景容量测试、备份恢复、迁移和回滚验证。

容量阈值不凭空指定。Wave 0 记录首发部署资源、代表性请求比例、数据规模及并发模型，由项目负责人确认后固化为版本化测试场景；在阈值确认和通过前，发布门禁保持未满足。

## 10. 执行与变更管理

- 限制在制品数量，同一时间只推进少量依赖清晰的切片；完成验收后再拉取下一批。
- 任务和提交标题包含稳定需求编号，例如 `feat(info): [I-02] 新增走访计划状态机`。
- 一个提交只完成一个可解释的逻辑变化，禁止跨需求大提交。
- 阻断出现时优先解决依赖，或切换到与该阻断无关的切片；不得合入无法验收的半成品接口。
- 正式需求矩阵持续更新状态和证据，不维护第二份完成度台账。
- 任何范围解释冲突以正式需求基线为准；变更基线必须单独评审并保留来源。

## 11. 最终上线判据

只有同时满足以下条件，首个全量版本才可发布：

- 42 项正式需求全部为“已完成”，实现和验收证据可定位。
- 单元、集成、契约、端到端和非功能门禁全部通过。
- 权限、行级数据范围、字段级保护、审计和敏感导出在真实链路验证通过。
- 支付、积分、消息、附件、外部接入等关键流程通过重复、乱序、超时、重试、补偿和人工恢复测试。
- 已确认的容量场景达标，且不存在未接受的高危安全漏洞。
- 数据备份恢复、数据库迁移和应用回滚完成演练。
- 生产配置、密钥、账号、网络暴露和日志脱敏审查通过。
- 运行手册、配置清单、监控指标、告警规则和发布检查表齐全。
- 不存在生产默认密钥、占位账号、公开敏感接口或未关闭的高风险项。

## 12. 主要风险及控制

| 风险 | 控制措施 |
| --- | --- |
| 全量首发范围大，反馈周期过长 | 按可验收垂直切片交付并持续关闭需求矩阵，不等待模块整体完成 |
| 过度追求单元测试导致真实链路缺陷 | Wave 0 先建立真实依赖和 gateway 冒烟链路，之后强制分层测试 |
| 横切能力后补造成多模块返工 | P2b、审计、消息、附件和外部安全在业务铺开前完成 |
| 跨服务双写和重试产生数据不一致 | 单一数据所有权、本地事务加 outbox、全局业务号幂等和可审计补偿 |
| TDD 退化为实现后补测试 | 保存 Red 失败原因和 Green 结果；无失败测试证据不得关闭切片 |
| 测试套件变慢阻碍反馈 | 快速、集成、系统、发布四级门禁；本地优先运行最小相关集合，CI 完整运行 |
| 无截止日期导致范围内细节无限扩张 | 只实现正式需求和上线门禁要求；新增能力必须先变更需求基线 |
